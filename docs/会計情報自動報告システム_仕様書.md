# 会計情報自動報告システム 仕様書（検討・設計ドラフト）

**文書の位置づけ**: 実装はローカル環境で行う前提とし、本書は利用検討からアーキテクチャ・機能要件・非機能要件までをまとめた設計用ドラフトである。税務・会計上の正否の保証は行わない。運用前に税理士等への確認を推奨する。

**想定利用者**: マネーフォワード クラウド会計・確定申告（以下「MF」）を利用し、登録メールが Gmail、請求書等がメールで届く事業者。

---

## 1. 背景と目的

### 1.1 背景

- 会計データの母艦として MF を利用している。
- 認証・通知の起点が Gmail であり、取引先からの請求書・領収書・明細メールが同じメールボックスに集約される。
- 「報告」を手作業で繰り返すと漏れ・遅延・フォーマット不統一が発生しやすい。

### 1.2 目的（「報告」の定義を明確化する）

本プロジェクトでいう「報告」は、次のいずれかまたは複合を指す。**導入前に優先順位を決めること**。

| 種別 | 例 | 主な受け手 |
|------|-----|------------|
| A. 内部経営報告 | 月次の売上・費用・キャッシュ、未処理件数 | 経営者・社内関係者 |
| B. 監査・役員向け | 四半期サマリー、証憑一覧、例外リスト | 取締役会、投資家 |
| C. 士業・顧問向け | MF 上の状況サマリー、未分類明細、添付の所在 | 税理士 |
| D. 税務申告用の下準備 | 申告期向けチェックリスト（MF は申告機能を持つが、自動で「申告書提出」までを意味しない） | 本人／士業 |

**推奨**: まず **A＋C** を自動化し、税務最終判断は人（または士業）が行う境界を仕様に明示する。

---

## 2. 現状資産と制約

### 2.1 既存システム

- **MF（クラウド会計・確定申告）**: 仕訳・証憑・連携口座等の正本とみなす。
- **Gmail**: 登録用メールおよび請求書等の受信。

### 2.2 制約・前提

- 実装・実行環境は **ローカル**（自社 PC または自社管理サーバ）。クラウド常駐は任意フェーズで検討可能。
- メールに含まれる個人情報・取引情報は **個人情報保護法・GDPR（該当時）** の観点で取り扱い方針が必要。
- MF への書き込み（仕訳登録等）は **誤仕訳リスク** があるため、初期は「取り込み候補の提示」または「ドラフト＋人の承認」を推奨。

## 3. 連携方式の比較（使用検討）

### 3.1 方式一覧

| 方式 | 概要 | メリット | デメリット・リスク |
|------|------|----------|-------------------|
| **① MF クラウド会計 API** | OAuth 2.0 で認可し、MF からデータ取得・仕訳登録等（提供機能の範囲内） | MF を正とした一貫した自動化が可能。確定申告版利用中でも API は追加料金なし（公式ガイド記載） | 実装は自社負担。公式による技術サポートは **有料公認メンバー（士業事務所）に限定**（2026年3月時点の案内）。API キー認可は非対応 |
| **② メール取り込みパイプライン** | Gmail API 等で対象メールを取得し、PDF／画像を保存・OCR・メタデータ抽出 | 請求書メール起点の自動化に直結 | 誤認識、なりすましメール、添付形式の多様さへの対応コスト |
| **③ MF 既存の金融・サービス連携** | 銀行・カード・決済等を MF 側で直接連携 | メールに依存しない取引の捕捉 | メール添付のみの証憑は別途必要 |
| **④ 手動エクスポート＋ローカル集計** | MF から CSV／帳票を定期出力し、ローカルで整形 | 実装が単純、権限設定が軽い | リアルタイム性が低い、人的運用が残る |
| **⑤ 外部 iPaaS（ノーコード）** | MF が案内する「外部サービス経由の API 利用」等 | 開発工数削減の可能性 | サブスク費用、データの外部預かり、ローカル実行との相性 |

### 3.2 推奨コンビネーション（ローカル実装向け）

1. **データの正**: MF（仕訳・マスタ・連携取引）。
2. **証憑の入口**: Gmail → ローカルに「未処理キュー」保存（原本 PDF／画像＋メールヘッダ）。
3. **自動化の軸**:
   - **読み取り中心**: ① MF API で残高・仕訳一覧・未確定項目を取得し、レポート生成。
   - **書き込みは段階導入**: まずは②で抽出した情報を **下書き JSON／CSV** にし、MF 手動取込 or 承認後 API 投稿。

- クラウド会計 API 概要・事前準備（アプリポータル、OAuth）: [マネーフォワード クラウド会計APIについて](https://biz.moneyforward.com/support/tax-return/guide/others/ot09.html)

## 4. システム構成（論理アーキテクチャ）

```
[Gmail] ──取得──► [ローカル: メール・添付インジェスト]
                        │
                        ▼
              [証憑ストア（暗号化ディスク推奨）]
                        │
                        ▼
              [抽出/OCR/ルールエンジン] ──► [取引候補（人間承認キュー）]
                        │
[MF クラウド会計 API] ◄──読取/（任意）書込──► [ローカル: 同期・レポート生成]
                        │
                        ▼
              [報告アーティファクト]
                - PDF / HTML / Markdown
                - スプレッドシート出力（任意）
                - （任意）Slack/メール通知はローカルから送信
```

### 4.1 コンポーネント責務

| コンポーネント | 責務 |
|----------------|------|
| メールインジェスト | ラベル／検索クエリで対象スレッドを増分取得、Message-ID で重複排除 |
| 証憑ストア | ファイルパス、ハッシュ、受信日時、差出人、MF 紐付けキー（後付け）を保持 |
| 抽出モジュール | PDF テキストレイヤー優先、無い場合は OCR。金額・日付・取引先名をヒューリスティック抽出 |
| MF クライアント | OAuth トークン管理、レート制限、リトライ、監査ログ |
| レポート生成 | 期間指定、科目別集計、例外（未分類・金額不整合・重複疑い）の一覧 |
| 設定 | YAML／環境変数。シークレットは OS シークレットストアまたは `.env`（git 除外） |

## 5. 機能要件

### 5.1 必須（MVP）

- **F-01**: 指定期間の MF データ取得（少なくとも「仕訳一覧相当」「残高／口座」の取得可能性は API 仕様に従う）。
- **F-02**: Gmail から対象メール＋添付をローカル保存し、一覧（CSV）を出力。
- **F-03**: 月次レポート（テンプレート）を生成し、経営者がそのまま共有できる形式（PDF 推奨）で出力。
- **F-04**: 実行ログ（いつ・何件・どの API を呼んだか）をローカルに残す。

### 5.2 推奨（Phase 2）

- **F-05**: OCR＋ルールで取引候補を生成し、承認 UI（ローカル Web でも可）で MF へ投稿。
- **F-06**: メール差出人ドメイン・件名パターンによる自動ラベリング。
- **F-07**: MF 側の「未処理」や例外条件と突合したアラート。

### 5.3 任意（Phase 3）

- **F-08**: 四半期・年次の拡張テンプレート、複数事業者（MF 事業所）対応。
- **F-09**: バックアップ用の増分アーカイブ（証憑＋メタデータ）。

## 6. 非機能要件

| 項目 | 目標 |
|------|------|
| 機密性 | ローカル保存データのディスク暗号化、バックアップ媒体の管理、リポジトリへの証憑混入禁止 |
| 可用性 | ローカル実行のため「スケジューラ（cron/Systemd）＋失敗時通知」で担保 |
| 保守性 | MF API・Gmail API の変更に備え、クライアントを薄くモジュール分割 |
| 監査性 | 人が承認した記録（誰が・いつ・何を）を残す |
| 法令・契約 | Google API・MF 利用規約の遵守。MF の **アプリポータル権限** は最小付与（公式注意事項に従う） |

## 7. 認証・権限（MF）

公式ガイドに基づく事前準備の要約:

1. [マネーフォワード クラウドのアプリポータル](https://app-portal.moneyforward.com/) の利用開始。
2. 管理コンソールの **全権管理** 権限が必要な操作がある。
3. アプリポータルでユーザーの **アプリ連携権限（クラウド会計・確定申告）** を付与。
4. 認可は **OAuth 2.0**（API キー不可）。
5. API 利用時の実効権限は **アプリポータル側の設定が優先**（MF 本体の権限と異なる点に注意）。

## 8. Gmail 連携方針

| 手段 | 用途 |
|------|------|
| Gmail API（OAuth、readonly 推奨） | ラベル付きメールの列挙・添付ダウンロード |
| 代替: 専用アドレスへ自動転送＋ローカル IMAP | API 審査負荷を下げたい場合（セキュリティ設計は別途） |

**推奨ラベル例**: `Accounting/Invoices`, `Accounting/Receipts`, `Accounting/NeedsReview`

## 9. データモデル（ローカル保存・論理）

最低限のエンティティ:

- **EmailMessage**: `message_id`, `thread_id`, `received_at`, `from`, `subject`, `labels[]`
- **Attachment**: `sha256`, `mime`, `local_path`, `parent_message_id`
- **ExtractionResult**: 抽出した `amount`, `tax`, `vendor_guess`, `doc_date`, `confidence`
- **MFLink**: `mf_journal_id` 等（API で取得できる識別子に合わせる）
- **ReportRun**: `period_start`, `period_end`, `artifact_paths[]`, `status`

---

## 10. レポート仕様（テンプレート項目案）

1. 表紙（期間、生成日時、データソースの版情報）
2. 損益サマリー（MF 上の集計に準拠）
3. 大口取引トップ N
4. 未分類・要確認リスト（件数・金額・証憑へのリンク）
5. メール取り込み状況（新規・未紐付け件数）
6. 付録: 実行ログサマリー

---

## 11. エラーハンドリング・運用

- MF・Google のトークン期限切れ時のリフレッシュ失敗 → 運用者通知。
- API レート制限 → 指数バックオフ。
- 重複メール・重複添付 → ハッシュで検知しスキップ。
- 定期ジョブの **ドライラン** モード（書き込みなしでレポートのみ）。

## 12. 技術スタック候補（ローカル実装）

実装者の習熟度で選択。要件を満たせば可。

| 層 | 候補 |
|----|------|
| 言語 | Python / Node.js / Go |
| MF クライアント | OpenAPI から生成、または手書きの薄い HTTP クライアント |
| スケジュール | cron、systemd timer、または OS 用タスクスケジューラ |
| OCR | Tesseract、またはクラウド OCR（ローカル方針と両立するなら「オプション」扱い） |
| レポート | HTML→PDF（Playwright / wkhtmltopdf 等）、または LaTeX |

## 13. リスク登録簿（抜粋）

| ID | リスク | 緩和 |
|----|--------|------|
| R-01 | OCR・抽出ミス | 置信度閾値、人の承認、MF 上で二重計上チェック |
| R-02 | メールなりすまし | 差出人検証、既知ドメインリスト、異常時隔離 |
| R-03 | API 仕様変更 | クライアントのバージョン固定と定期スモークテスト |
| R-04 | 税務判断の誤り | 自動化は「下準備」に限定し、最終確認を士業または本人に明示 |

## 14. マイルストーン

1. **M0**: 「報告」の受け手と KPI（例: 月次 3 営業日以内）を確定。
2. **M1**: MF API の読み取りのみでレポート原型。
3. **M2**: Gmail 取り込み＋証憑ストア＋レポートへの「未紐付け」セクション。
4. **M3**: （任意）承認付き仕訳投稿。
